### Diagnosis

**What's working:**
- Strong architectural identity — barebone + theme separation is a genuine differentiator
- Good forms integration where it matters (checkbox, toggle, radio, date-picker)
- Reasonable a11y baseline on most controls
- Clean secondary-entrypoint structure for tree-shaking

**What's not working:**
- **Zero test coverage** across the entire library — this is the #1 quality risk and blocks any credible 1.0 claim
- CSS cascade compliance is inconsistent — only Input fully follows the documented `:where()` + `@layer` pattern
- Spinner uses `insertBefore` instead of `appendChild`, violating the cascade ordering rule
- Modal lacks focus trap — a critical a11y gap for dialogs
- `todo.md` is fully checked off with no forward-looking backlog

### Recommendations

**1. [P0] — Complete the form primitive set before adding anything else**

The fastest path to adoption is a developer being able to build a complete form with Bloc-UI. Today they can do Input, Checkbox, Radio, Toggle, and Date Picker — but they hit a wall at **Select** and **Textarea**. These are in every form. Without them, developers can't go all-in on Bloc-UI and end up mixing libraries.

- **Textarea** is a near-zero-effort win — it's a `textarea[blocTextarea]` directive following the exact same pattern as `input[blocInput]`. Ship it this week.
- **Select** is the highest-leverage single component to build. It needs a dropdown panel (reusable as a popover primitive later), keyboard navigation, and CVA integration. Medium effort, massive payoff.

**2. [P0] — Ship Tooltip**

Tooltip is the most universally needed UI primitive after buttons and inputs. It's low effort (a directive that positions a floating element), high visibility, and signals library maturity. Every competitor has it.

**3. [P0] — Add baseline tests before any new components**

Zero test coverage is an existential risk. One accidental breaking change in a release and trust evaporates. Before building new components:
- Add CVA round-trip tests for checkbox, toggle, radio, date-picker triggers
- Add basic render + interaction tests for all existing components
- This also establishes test patterns that new components follow from day one

**4. [P1] — Enforce CSS cascade compliance across all existing components**

Button is the most visible component and doesn't follow the `:where()` / `@layer` pattern at all. Spinner uses `insertBefore` instead of `appendChild`. These inconsistencies undermine the library's core promise. Remediate before shipping new components.

**5. [P1] — Add focus trap to Modal**

A dialog without focus trap fails WCAG 2.1 SC 2.4.3 (Focus Order). This is a blocking a11y bug, not a feature request. Should be fixed before any new component work.

**6. [P1] — Ship Progress, Alert, Badge, and Pagination**

After the P0 items, this tier rounds out the library:
- **Progress** + existing Spinner = complete loading pattern
- **Alert** + existing Toast = complete feedback pattern
- **Badge** is a tiny component, quick win
- **Pagination** + existing Table = usable data pattern

### Trade-offs Acknowledged

| Choosing NOT to... | Because... |
|---------------------|-----------|
| Build a Combobox/Autocomplete now | It depends on a popover primitive that doesn't exist yet. Build Tooltip + Select first, extract shared positioning logic, then Combobox becomes incremental |
| Ship a Sidebar or Navigation component | These are layout-heavy, app-architecture-opinionated components that conflict with our "zero opinion" identity. Defer until demand is proven |
| Chase PrimeNG's 80-component breadth | Fewer components done well beats many done poorly. We compete on lightweight + accessible + themeable, not component count |
| Build a Carousel | Low leverage for a component library. It's a solved problem (Swiper, etc.) and not core to form-heavy business apps |
| Add i18n infrastructure now | Premature. Only Date Picker has localizable strings. Revisit when component count justifies the abstraction |
